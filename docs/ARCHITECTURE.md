# 架构设计文档

## 设计哲学：声明式优于硬编码

本仓库的核心设计选择是**用 markdown + JSON schema 替代 Python 代码**作为智能体的"源代码"。

一个传统 AI 应用通常由一个 Python 脚本定义它的行为流程。本仓库则相反——`AGENT.md` 是主程序，`skills/*/SKILL.md` 是函数库，`schemas/*.schema.json` 是类型系统。Python 脚本仅在 LLM 本身无法完成的操作（加载模型、批量文件处理）时作为兜底存在。

这样设计的理由：
- **可审计性**：任何人都能阅读 AGENT.md 和 SKILL.md 来完全理解智能体的行为逻辑，无需阅读代码
- **可修改性**：修改行为只需编辑 markdown 文本，不需要编程技能
- **平台无关性**：不绑定特定 LLM 客户端或 API。只要一个平台能加载 AGENT.md 作为系统提示、能调用网页检索工具，就能激活这个智能体
- **演进性**：当 LLM 能力提升时（如更好的推理、更强的工具调用），智能体的能力同步提升，无需改代码

## 五阶段工作流

```
用户提交材料 (inputs/)
        │
        ▼
┌─────────────────────────────────────────────┐
│ 阶段一：用户数据输入分析                      │
│ skill: stage1-parse-input                   │
│ 输入: 非结构化材料（文本/PDF/图像）           │
│ 输出: event_skeleton.json                   │
│ 动作: 抽取 + 四类区分（事实/断言/线索/缺失）   │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│ 阶段二：事件推理                             │
│ skill: stage2-reason-events                 │
│ 输入: event_skeleton.json                   │
│ 输出: hypotheses.json (3-5 条)              │
│ 动作: 四问框架 + 先验权重 + 因素标注          │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│ 阶段三：归因分析和证据检索                    │
│ skill: stage3-retrieve-evidence             │
│ 输入: hypotheses.json + event_skeleton      │
│ 工具: tool-logistics / tool-materials       │
│       tool-projects / tool-policy           │
│       tool-general-web (兜底)                │
│ 输出: evidence_log.json                     │
│ 动作: N×4 强制检索 + 交叉验证                 │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│ 阶段四：因果综合与影响量化                    │
│ skill: stage4-synthesize-causal             │
│ 输入: evidence_log.json + hypotheses        │
│ 输出: causal_chain.md + impact_estimates    │
│       + uncertainty_notes.md               │
│ 动作: [Ex]引用 + 区间估计 + 反思              │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│ 阶段五：写入案例记忆库                        │
│ skill: stage5-write-memory                  │
│ 输入: runs/<case_id>/ 全量输出              │
│ 输出: memory/cases/<case_id>/ + INDEX.md    │
│ 动作: 迁移 + 关联 + 索引 + 用户摘要          │
└─────────────────────────────────────────────┘
```

## 约束规则的来源

AGENT.md 第 2 节的六条工作原则（P1–P6）直接来源于论文的理论推导：

| 原则 | 论文来源 | 含义 |
|:---|:---|:---|
| P1 强制跨因素检索 | §2.1.1 涟漪效应 + 四重影响因素；§2.1.2 证据完整性原则 | BESS 中断跨四类因素传导，缺少任一因素的检索即归因不完整 |
| P2 无记录不主张 | §4.3.4 "标注各环节的证据来源" | 因果链中每个事实性主张必须可追溯到 evidence_log 条目 |
| P3 先生成假设再寻证据 | §4.2.2 "避免在尚未调用工具检索之前就贸然给出单一结论" | 降低 LLM 幻觉风险的结构性措施 |
| P4 区间估计而非伪精确 | §4.3.4 "不强求生成精确数字" | 非结构化数据的稀疏性与公开证据的间接性决定了估计只能是区间 |
| P5 平实语言 | §1.3 "面向供应链经理" | 避免学术术语，用行业管理者能理解的语言输出 |
| P6 五阶段不可跳跃 | §4.2 处理流程的整体设计 | 保证分析的可追溯性与可审计性 |

## 工具集的理论推导

四类专用工具不是从"可用的 API 列表"中凭经验挑选的，而是从论文 §2.1.1 的 BESS 四重影响因素出发逐一推导的。每类因素各对应一类有明确时间/地域/权威性的外部证据源：

```
物流因素 → 航运指数 + 港口效率 + 航线运力 → tool-logistics
原材料因素 → 关键矿物现货/期货价格 → tool-materials
合同与项目因素 → 财报风险披露 + 行业统计 + 延期报道 → tool-projects
合规与政策因素 → 官方公报 + 海关执法 + 监管公告 → tool-policy
```

这种从理论结构推导工具集的设计逻辑，是本研究区别于既有工具按功能模块堆叠风险数据源的关键贡献之一（论文 §2.1.2）。

## 案例记忆库的数据模型

论文 §4.4 定义了案例条目（Case Entry）的完整数据模型。本仓库以 `schemas/case_entry.schema.json` 作为该模型的机器可读定义，`memory/cases/<case_id>/case_entry.json` 是其实例。

记忆库采用"结构化元数据 + 可选的向量索引"的混合形式——前者支持精确字段查询（SQL 式过滤：某时间窗口、某物料、某供应商），后者支持语义检索（"类似的中断事件有哪些"）。
